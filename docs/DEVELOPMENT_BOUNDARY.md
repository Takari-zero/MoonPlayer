# Moon播放器 开发边界

## 1. 修改原则

每轮开发必须遵守：

- 每轮只改一个模块。
- 未经用户确认，不跨模块修改。
- 修改前必须说明范围。
- 修改后必须说明改了哪些文件。
- 完成功能开发后必须运行 `.\gradlew.bat :app:assembleDebug`。
- 编译失败必须输出完整错误。
- 用户要求只写文档时，不得修改业务代码。
- 用户要求暂停功能开发时，必须停止功能改动。

## 2. 模块边界

### 视频模块

只允许修改：

- `VideoLibraryScreen`
- `FolderScreen` 的视频模式
- `VideoPlayerScreen`
- 视频相关 model / util / state
- 必要的视频扫描、删除、隐藏记录逻辑

不得顺手修改音乐、小说、设置模块。

### 音乐模块

只允许修改：

- `AudioLibraryScreen`
- `AudioPlayerScreen`
- `MusicPlaybackService`
- 音乐相关 util / state
- 音乐后台播放、队列、通知、隐藏记录逻辑

不得顺手修改视频、小说模块。

### 小说模块

只允许修改：

- `BookLibraryScreen`
- `BookListenScreen`
- TTS 相关文件
- TXT reader 相关文件
- book progress 相关文件
- 小说导入、删除、持久化相关逻辑

不得顺手修改视频、音乐模块。

### 我的 / 设置模块

只允许修改：

- `ProfileScreen`
- `SettingsScreen`
- DataStore 对应设置项
- App 信息和数据管理入口

不得把设置页重新放回底部导航。

### 共享文件

`MainActivity`、DataStore、导航和共享状态只有在当前模块确实需要路由、状态或持久化接入时才允许修改。

修改共享文件时必须说明原因和影响范围。

## 3. 禁止 Codex 自作主张

未经用户明确批准，不得：

- 新增用户未要求的功能。
- 修改 App 名称。
- 修改 App 图标。
- 修改包名。
- 修改 `applicationId`。
- 引入 Room。
- 引入 Hilt。
- 引入 ViewModel。
- 改其它模块做“顺手优化”。
- 删除已有功能。
- 把占位功能做成假成功。
- 申请 `MANAGE_EXTERNAL_STORAGE`。
- 添加登录、云同步、广告或联网推荐。

## 4. 功能未实现时的处理

未实现功能必须明确反馈：

- 使用 Toast 说明“后续实现”。
- 不得点击无反应。
- 不得假装执行成功。
- 危险操作失败时必须说明原因。
- 删除失败时必须保留原数据，不得误删 UI 状态。

## 5. Git 和验证

每轮完成后必须说明：

- 修改了哪些文件。
- 新增了哪些文件。
- 保留了哪些原功能。
- 是否运行 `.\gradlew.bat :app:assembleDebug`。
- 编译结果。

常规流程：

1. 完成单轮修改。
2. 运行 `.\gradlew.bat :app:assembleDebug`。
3. `BUILD SUCCESSFUL` 后交给用户测试。
4. 用户测试通过后再提交 Git。

如果用户明确要求只写文档，可以不编译，但必须说明本轮没有改代码。

## 6. 优先级

开发优先级：

1. 稳定性
2. 已有功能不回退
3. UI 收口
4. 新功能
5. 细节动画

原则：

- 先单模块闭环，再做下一个模块。
- 先修 bug，再做视觉优化。
- 先完成核心链路，再补充高级功能。

## 7. 当前开发阶段

当前阶段：

1. 视频模块收口
2. 音乐模块收口
3. 小说模块收口
4. 设置 / 发布准备

在视频模块收口完成前，不应主动扩展音乐或小说新功能。

## 8. 每轮交付清单

每轮交付时应回答：

- 本轮改了哪些文件？
- 新增了哪些文件？
- 是否跨模块？
- 是否保留原功能？
- 是否编译？
- 如果未编译，原因是什么？
- 下一步建议是什么？
